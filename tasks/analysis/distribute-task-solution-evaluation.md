# 任务分发方案评估

## 背景

Capture 的 Local Task Center 需要将任务分发到多台机器上执行，本文档评估两种可能的方案。

## 核心需求

1. 在一台主控机器上统一调度任务
2. 任务下发到多台 worker 机器本地执行
3. 每台 worker 机器有唯一标识（机器 ID）
4. 所有操作均在 git repo 中进行
5. 有状态反馈机制，能看到执行进度
6. 最好能支持交互式 terminal

## 方案 1：远程控制 (Remote Control)

### 机制

利用 RustDesk 等远程桌面/控制工具，让人类"代为操作"远端机器：
1. 主控机器打开多台 worker 的远程桌面
2. 人为将任务发给远端执行

### 优势

- 实现最简单，几乎零开发

### 致命缺陷

- 需要人工介入，无法做到真正的任务分发自动化
- 每次任务都需要人操作，无法后台运行
- 无法与 Capture 系统联动，任务状态无法自动回流
- 扩展性差，机器多了人成为瓶颈
- 无法做执行进度追踪

### 结论

**不满足需求**。Remote Control 解决的是"人如何远程操作电脑"的问题，不是"如何让机器自动执行任务"的问题。

---

## 方案 2：飞书 Bot + 本地 Daemon（推荐）

### 机制

每台机器运行一个本地 Daemon，通过飞书 Bot 接收任务指令并执行：

1. 主控机器的 Capture 记录任务和目标机器
2. Capture 通过飞书 Bot 向目标机器的 Bot 发送指令
3. 每台机器的 Daemon 监听飞书消息，收到后解析并本地执行
4. 执行结果通过 Bot 回调或主动推送反馈给主控

### 关键组件

| 组件 | 职责 |
|------|------|
| Master Capture | 记录任务 → 机器的映射，下发指令 |
| Feishu Bot (per machine) | 每台机器一个 Bot，作为消息入口 |
| Local Daemon | 运行在每台 worker 上，监听 Bot 消息，执行任务，回传结果 |
| Message Protocol | 任务 ID、机器 ID、仓库路径、执行命令、状态回调 |

### 优势

- 与 Capture 现有飞书 Bot 积累完全复用
- 纯消息驱动，后台自动化执行，无需人工介入
- 每台机器独立 Bot，故障隔离，单点故障不影响其他机器
- 天然支持多机器横向扩展

### 劣势

- 需要为每台 worker 机器配置独立的飞书应用（Bot）
- 需要设计可靠的消息协议和回调机制
- Daemon 的安装和配置有一定成本

---

## 严格结论

### 推荐：方案 2（飞书 Bot + Daemon）

理由：

1. **真正的自动化**：机器接收任务后自动执行，不需要人介入
2. **与 Capture 现有体系一致**：Capture 已有飞书 Bot 实现，可直接复用
3. **轻量级**：不需要远程桌面、屏幕录制等重量级依赖
4. **可扩展**：新增机器只需部署一个 Daemon + 注册一个 Bot

### 方案 1 不适合的原因

Remote Control 类工具（RustDesk 等）解决的是"人远程操作电脑"的问题，而不是"机器自动执行任务"的问题。两者的本质区别：

| | Remote Control | Bot + Daemon |
|--|--|--|
| 触发方式 | 人 | 消息 |
| 执行方式 | 人在远端操作 | 机器自动执行 |
| 与 Capture 联动 | 差 | 好 |
| 后台运行 | 否 | 是 |
| 状态回流 | 无 | 可设计 |

---

## 建议实施路线

**Phase 1（快速验证）**：
- 单机版：Capture → 飞书 Bot → 本地 Daemon 执行shell 命令
- 验证消息协议、执行回调机制

**Phase 2（多机扩展）**：
- 支持多机器注册
- 每台机器独立 Bot
- 统一调度界面（CLI/TUI）

**Phase 3（增强）**：
- Web dashboard 查看各机器执行状态
- 支持 terminal 交互（通过 Bot 会话）
