# Brutal Reality Check

> **残酷的现实检查：在产品/想法早期识别妄想与可行的系统性方法**

---

## 1. 概述

### 1.1 这是什么？

**Brutal Reality Check** 是一套系统化的思维框架，用于在产品或想法的早期阶段，从多个维度进行严格审视，识别：

- ❌ **妄想 (Delusion)**: 解决不存在的问题、面向不存在的市场
- ⚠️ **部分可能 (Partially Viable)**: 有 value 但需要调整方向或缩小范围
- ✅ **可行 (Viable)**: 真实需求 + 清晰路径 + 可盈利

### 1.2 为什么需要？

**常见问题：**
- 技术人员容易陷入"解决方案先行"（先想酷的技术，再找问题）
- 概念堆砌（用新词包装旧想法，听起来很酷但定义不清）
- 极客中心主义（从自己视角出发，忽视大众用户）
- 混淆"能"与"会"（技术可行 ≠ 用户会采用）

**目标：**
在早期（投入大量资源前）识别问题，避免构建"解决不存在问题的产品"。

## 2. 核心框架：三层检查

### Layer 1: 逻辑层检查 (Logical Validation)

检查想法的内在逻辑一致性。

```markdown
## 执行步骤

1. **拆解逻辑链条**
   - 将核心主张拆解为：前提 → 推论 → 结论
   - 示例：（本地必须做加工）
     * P1: IM 界面有局限
     * P2: 深度加工需要复杂状态
     * C:  因此本地必须做加工

2. **检查每一步的有效性**
   - [ ] 前提是否成立？
   - [ ] 推论是否必然？
   - [ ] 是否有隐含假设？
   - [ ] 是否忽略了替代方案？

3. **识别概念冲突**
   - 主张 A vs 主张 B 是否矛盾？
   - 目标 vs 方案是否一致？
   - 理想 vs 现实是否脱节？

## 输出格式

| 主张 | 逻辑有效性 | 隐含假设 | 概念冲突 |
|------|-----------|---------|---------|
| 本地必须加工 | ⚠️ 部分有效 | 数据必须在本地 | 忽略了网页/App |
```

### Layer 2: 前提层检查 (Premise Validation)

检查想法依赖的前提假设是否成立。

```markdown
## 关键前提检查清单

### P1: 用户想要这个吗？
- [ ] 是"需要"(Need) 还是"想要"(Want)？
- [ ] 用户当前如何解决？
- [ ] 现有方案的问题有多痛？
- [ ] 用户愿意为此付费吗？

**警示信号：**
- ❌ "用户还没意识到他们需要"
- ❌ "教育市场"
- ❌ "改变用户习惯"

### P2: 市场规模足够大吗？
- [ ] 目标用户基数？
- [ ] 层层筛选后的转化率？
  - 知道产品 → 感兴趣 → 试用 → 付费
- [ ] 计算：用户数 × ARPU × 转化率 = 收入
- [ ] 是否达到盈利门槛？

**警示信号：**
- ❌ 层层筛选后用户 < 1000
- ❌ 年收入 < 成本

### P3: 技术/方案是最佳吗？
- [ ] 现有替代方案是什么？
- [ ] 你的方案好 10x 还是好 10%？
- [ ] 转换成本有多高？
- [ ] 为什么不用现有方案组合？

**警示信号：**
- ❌ 比现有方案更复杂
- ❌ 好 10% 但转换成本高
- ❌ 现有免费方案已满足

### P4: 时机对吗？
- [ ] 市场准备好了吗？
- [ ] 基础设施成熟了吗？
- [ ] 是趋势还是 fad？

**警示信号：**
- ❌ "太早了"（市场未成熟）
- ❌ "太晚了"（已被巨头占据）
```

### Layer 3: 冲突层检查 (Conflict Detection)

识别想法内部的矛盾和不一致。

```markdown
## 常见冲突模式

### 冲突 1: 目标 vs 方案
- **模式**: 声称要简化，但方案更复杂
- **检查**: 数一下用户需要操作的步骤
- **示例**: "让用户更高效" → 但需要学习 10 个新概念

### 冲突 2: 理想 vs 现实
- **模式**: 理想状态描述很好，但现实路径不清
- **检查**: 从现状到理想的具体步骤？
- **示例**: "Sovereign Individual"理想很好，但99%人不想要

### 冲突 3: 能 vs 会
- **模式**: 技术上"能"做到，但用户"会"去做吗？
- **检查**: 能力赋予 ≠ 行为改变
- **示例**: AI 让个人"能"拥有机构能力，但大多数人"不会"去用

### 冲突 4: 说 vs 做
- **模式**: 说的和做的不一致
- **检查**: 观察实际行为 vs 声称的偏好
- **示例**: 说想要"信息主权"，但行为上选择算法 feed

### 冲突 5: 通用 vs 特定
- **模式**: 声称通用解决方案，但实际只解决特定问题
- **检查**: 去掉 buzzwords 后还剩什么？
- **示例**: "革命性平台" → 实际是个 todo app

## 输出格式

| 冲突类型 | 是否存在 | 具体表现 | 严重程度 |
|---------|---------|---------|---------|
| 目标-方案 | ✅ 是 | 声称简化但复杂 | 🔴 高 |
| 能-会 | ✅ 是 | 能做到但用户不用 | 🟡 中 |
```

## 3. 执行流程

### Step 1: 写下核心主张 (5 min)

用一句话描述你的想法：
```
我们为 [目标用户] 提供 [解决方案],
帮助他们解决 [问题],
通过 [独特方式],
从而获得 [价值]。
```

### Step 2: 逻辑层检查 (15 min)

执行 Layer 1 的检查：
- 拆解逻辑链条
- 检查每一步
- 识别概念冲突

**输出**: 逻辑有效性评估

### Step 3: 前提层检查 (20 min)

执行 Layer 2 的检查：
- 检查 4 个关键前提
- 计算市场规模
- 列出替代方案

**输出**: 前提假设验证表

### Step 4: 冲突层检查 (15 min)

执行 Layer 3 的检查：
- 检查 5 种冲突模式
- 寻找不一致之处
- 识别隐藏矛盾

**输出**: 冲突检测表

### Step 5: 综合判决 (10 min)

根据三层检查结果，做出判决：

```markdown
## 判决矩阵

| 检查层 | 结果 | 证据 |
|-------|------|------|
| 逻辑层 | ✅/⚠️/❌ | ... |
| 前提层 | ✅/⚠️/❌ | ... |
| 冲突层 | ✅/⚠️/❌ | ... |

## 最终判决

- [ ] ❌ 妄想 (Delusion): 根本性错误，建议停止
- [ ] ⚠️ 部分可能 (Partially Viable): 需要重大调整
- [ ] ✅ 可行 (Viable): 可以 proceed，但需验证

## 如果继续，需要验证：
1. 
2. 
3. 
```

## 4. 示例

### 示例 1: 失败的检查

**想法**: 本地 AI Agent 加工层

**逻辑层检查:**
| 主张 | 有效性 | 问题 |
|------|--------|------|
| IM 不适合加工 | ✅ 有效 | IM 确实有局限 |
| 因此本地必须加工 | ⚠️ 无效 | 忽略了网页/App 选项 |

**前提层检查:**
- P1 用户想要？❌ 现有工具已满足
- P2 市场大？❌ <0.01% 人群
- P3 方案最佳？❌ 比现有方案复杂

**冲突层检查:**
- 目标简化 vs 方案复杂 ✅ 冲突
- 能 vs 会 ✅ 冲突

**判决**: ❌ **妄想** - 解决不存在的问题

---

### 示例 2: 通过的检查

**想法**: AI 驱动的学术文献综述工具

**逻辑层检查:**
| 主张 | 有效性 | 问题 |
|------|--------|------|
| 研究者需要读大量文献 | ✅ 有效 | 真实痛点 |
| AI 可以帮助筛选 | ✅ 有效 | 技术可行 |
| 因此有市场 | ✅ 有效 | 逻辑成立 |

**前提层检查:**
- P1 用户想要？✅ 真实痛点
- P2 市场大？⚠️ 小众但付费能力强
- P3 方案最佳？✅ 比手动好 10x

**冲突层检查:**
- 无明显冲突 ✅

**判决**: ✅ **可行** - 垂直领域真实需求

## 5. 常见陷阱

### Trap 1: 概念堆砌 (Buzzword Stacking)

```
❌ "我们用 AI 驱动的去中心化 Web3 平台，
     赋能 Sovereign Individual 在元宇宙中..."

检查方法:
1. 去掉所有 buzzwords
2. 还剩什么？
3. 如果是空话 → 陷阱

✅ "我们帮助律师自动生成合同初稿"
```

### Trap 2: 技术乐观主义

```
❌ "AI 可以做到 X，所以用户一定会用"

检查方法:
1. 技术可行 ≠ 用户采用
2. 考虑学习成本
3. 考虑切换成本
4. 考虑维护成本
```

### Trap 3: 极客中心主义

```
❌ "我想要这个功能，所以其他人也想要"

检查方法:
1. 你是目标用户吗？
2. 调查 10 个非技术人员
3. 他们关心吗？
```

### Trap 4: 解决方案先行

```
❌ "我们有很酷的技术，找问题来应用"

检查方法:
1. 先描述问题，不提解决方案
2. 问题真实存在吗？
3. 人们愿意付费解决这个问题吗？
4. 然后再看技术是否匹配
```

## 6. 何时使用

### ✅ 使用场景

- 产品想法早期（投入资源前）
- 技术选型决策
- 商业计划评估
- 方向调整考虑

### ❌ 不适用场景

- 已经验证的产品迭代
- 纯技术研究
- 艺术创作
- 个人学习项目（不需要商业化）

## 8. 总结

**Brutal Reality Check** 的核心价值：

1. **早期识别**：在投入资源前发现问题
2. **系统全面**：三层检查覆盖逻辑、前提、冲突
3. **避免偏见**：强制从不同角度审视
4. **决策依据**：提供结构化判断框架

**记住：**
- 被 BRC 判定为"妄想" ≠ 想法没价值
- 可能是时机不对、市场未成熟、或需要调整方向
- 记录思考过程本身就有价值

**最终目标：**
不是杀死想法，而是确保构建的是**解决真实问题的产品**。
