# 任务分发方案完整评估与实施设计

## 背景

Local Task Center 的一个关键目标，是把任务稳定地下发到不同机器执行，并且能够知道：

- 哪个任务被发到哪台机器
- 在哪个仓库执行
- 当前是否在运行
- 执行结果是否已经回流
- 后续是否可以扩展到可视化终端和 Web 观察面

当前需要评估两条路线：

1. 远程控制方案
2. 飞书 Bot + 本地 Daemon 方案

目标是轻量级、个人使用、尽快落地。

## 评估结论

最终建议优先采用 **方案 2：飞书 Bot + 本地 Daemon**。

原因不是它“最理想”，而是它同时满足以下条件：

- 能做到自动化分发，而不是人工远程操作
- 能复用当前项目已有的飞书 Bot 能力
- 架构简单，易于从单机演进到多机
- 更适合后续补上状态回流、Web 面板、终端追踪

方案 1 可以作为临时辅助工具，但不适合作为主实现路线。

## 方案 1：远程控制

### 方案描述

主控机器通过 RustDesk 一类远程控制工具连接多台机器，然后由人类在远程桌面里手动触发任务执行。

### 能解决的问题

- 快速连接远端机器
- 适合临时维护、人工排障、手动执行
- 初期几乎不用开发

### 不能解决的问题

- 任务不是自动分发，而是人工分发
- 无法形成稳定的任务协议
- 无法自动回写任务执行状态
- 无法可靠扩展到 4-5 台机器以上
- 难以形成与 Task Center 一致的数据流

### 对 RustDesk 的判断

RustDesk 能解决“远程看到并操作机器”的问题，但不能直接解决“任务分发中心”的问题。

如果强行基于远程控制来做：

- 仍然需要额外的任务协议
- 仍然需要每台机器本地有执行代理
- 仍然需要状态回流和任务登记

也就是说，真正要实现目标，最终还是会回到“每台机器有一个本地服务/代理”的架构。

### 结论

方案 1 适合作为运维辅助，不适合作为主路径。

## 方案 2：飞书 Bot + 本地 Daemon

### 方案描述

每台机器部署一个本地 Daemon，并用独立的飞书 Bot 或独立的飞书接收通道作为任务入口：

1. 主控侧选择任务和目标机器
2. 主控侧发送任务消息
3. 目标机器的 Daemon 接收消息
4. Daemon 在指定 repo 中执行
5. 执行结果回传主控侧

### 优势

- 是真正的自动化任务分发
- 与当前项目已有飞书能力一致
- 每台机器天然是独立 worker
- 容易做状态回流和心跳监控
- 后续可以补 Web 和 terminal 观察面

### 风险

- 需要设计消息格式
- 需要解决幂等、重试、回调可靠性
- 需要管理机器身份和工作目录

### 结论

方案 2 是最适合当前项目的主路径，也是最快能做出 MVP 的方案。

## 两个方案的严格对比

| 维度 | 方案 1：远程控制 | 方案 2：Bot + Daemon |
|------|------------------|----------------------|
| 自动化能力 | 低 | 高 |
| 开发成本 | 低 | 中 |
| 扩展到多机 | 差 | 好 |
| 状态回流 | 弱 | 强 |
| 与 Capture 现有能力复用 | 低 | 高 |
| 后续接 Web/TUI | 弱 | 强 |
| 适合作为长期架构 | 否 | 是 |

## 推荐架构

### 核心角色

#### 1. Master

主控侧，负责：

- 选择任务
- 选择目标机器
- 生成分发消息
- 接收执行回调
- 更新任务状态

#### 2. Worker Daemon

每台机器上的常驻服务，负责：

- 监听任务消息
- 校验目标仓库
- 在指定目录执行命令
- 回传执行结果
- 定时汇报心跳

#### 3. Message Channel

优先使用飞书消息作为控制通道，后续可扩展成：

- 飞书消息下发
- HTTP 回调上报
- WebSocket 长连接

## 建议的数据结构

### 下发消息

```json
{
  "type": "task_dispatch",
  "task_id": "TASK-00001",
  "machine_id": "mac-mini-01",
  "repo": "/Users/patrick/workspace/project-a",
  "worktree": "feature/task-center",
  "command": "capture-worker run TASK-00001",
  "source": "capture-master"
}
```

### 结果消息

```json
{
  "type": "task_result",
  "task_id": "TASK-00001",
  "machine_id": "mac-mini-01",
  "status": "success",
  "exit_code": 0,
  "summary": "completed",
  "started_at": "2026-04-04T10:00:00Z",
  "completed_at": "2026-04-04T10:05:00Z"
}
```

## 实施计划

### Phase 1：最小可用版本

目标：先把“能分发、能执行、能回传”跑通。

范围：

- 单台 worker
- 固定 repo
- 固定命令模板
- 飞书消息下发
- 执行结果写回主控

交付物：

- `capture dispatch send`
- `capture worker daemon`
- 简单的任务回传与状态更新

### Phase 2：多机版本

目标：支持 4-5 台机器的稳定分发。

范围：

- 机器注册表
- 每台机器独立 machine_id
- 每台机器默认 repo 列表
- 心跳状态
- 失败重试

交付物：

- worker registry
- machine status
- retry / timeout 机制

### Phase 3：观察与交互

目标：把执行阶段变得可观察。

范围：

- 任务在哪台机器上执行
- 当前日志摘要
- 最近活跃时间
- Web 查看 daemon 状态
- TUI 查看机器视图

交付物：

- 机器状态看板
- 执行摘要视图
- review 队列

## 为什么它是“最快实现”的方案

如果只看“今天能不能让几台机器跑起来”，远程控制看起来最省事。

但如果看的是“最快做出一个真正可持续的任务分发中心”，答案反而是 Bot + Daemon。

原因是：

1. 远程控制几乎没有沉淀系统能力
2. Bot + Daemon 从第一天开始就在积累可复用的协议和架构
3. 后续 Web、TUI、状态回流、日志追踪都能沿着同一条链路继续扩展

所以更准确地说：

- **远程控制最快拿来临时用**
- **Bot + Daemon 最快做成真正产品**

## 最终建议

建议采用“双轨策略”：

1. 正式方案用 **Bot + Daemon**
2. 临时排障和人工接管时保留 **远程控制**

也就是：

- 把远程控制当“人工 fallback”
- 把 Bot + Daemon 当“产品主架构”

这样最符合轻量级、个人使用、快速落地三个目标。
